Detailed Analysis

This week's technical challenge was centered on the editor component of our app. While testing functionality, it became apparent that we had to goals in conflict with each other: first, to style the user's input based on the tone analysis provided by Watson, and second, to prevent the malicious insertion of html markup into our persistence layer. The initial, and safest solution involved stripping the user's input of tags after every keystroke. This had the unfortunate consequence of making it impossible to actually type coherently into the editor, requiring text be copy-pasted instead.

With my second attempt, I remained with my decision to read and write the editor's text value directly from the associated version on state. In order to keep our resilience to script injection, I moved the tag stripping logic into the methods that handled saving and analyzing the text. This iteration was the worst of them all, not actually solving the bugs created by the original design, while introducing additional bugs related to the storing of marked up text in the app state. Highlighting became unpredictable, as sentences were wrapped in fresh span tags after every keystroke. Additionally, it became possible to completely destroy the layout of the site by pasting in arbitrary HTML.

The final solution involved a completely different understanding of the needs of each actor in the "editor story." I realized that it was absolutely necessary to keep on hand the fully marked-up text for the version being edited. Just as important was to keep the stripped representation up to date in the version state. Having a single representation that was read from and written to on every key stroke was simply unfeasible. I employed a technique I now know as "denormalization," and created a new state variable to store the editorText, which contained all markup created by our app, as well as any arbitrary HTML the user might decide to enter. Behind the scenes, the text was stripped of its tags and saved to the current version. Whenever a user elected to analyze or save the version being edited, the stripped representation was provided to the server. Lastly, when the user loaded a version or requested an analysis, the editor's contents were clobbered and replaced by markup generated from the pristine text stored on state.

The result is an editor that works as it should during normal editing operation, and looks like we want it to when it's time to analyze.
